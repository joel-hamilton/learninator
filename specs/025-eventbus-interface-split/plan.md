# Implementation Plan: EventBus Interface Split

**Branch**: `025-eventbus-interface-split` | **Date**: 2026-06-19 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/025-eventbus-interface-split/spec.md`

## Summary

Split the monolithic `EventBus` interface in `src/ai/events.ts` into two focused interfaces: `ToolEventBus` (mission-scoped tool events with `subscribe`/`emit`) and `WorkflowEventBus` (user-scoped workflow events with `subscribeUser`/`emitUser`). The `createEventBus()` factory continues returning a single object satisfying both interfaces. All consumers (`conversationLoop`, `WorkflowStateManager`, `LessonGenerator`, `missionChatService`, home SSE route) are updated to accept only the bus interface they actually use. This is a pure structural refactoring with zero runtime behavior change — only TypeScript types and constructor/dependency signatures change.

## Phase 1: Design & Contracts

### Data Model

Two TypeScript interfaces (type-level only, no runtime):

- **ToolEventBus**: `{ subscribe(missionId: number, cb: ToolEventCallback): () => void; emit(missionId: number, event: ToolEvent): void }`
- **WorkflowEventBus**: `{ subscribeUser(userId: number, cb: WorkflowEventCallback): () => void; emitUser(userId: number, event: WorkflowEvent): void }`

`createEventBus()` return type changes from `EventBus` to `ToolEventBus & WorkflowEventBus`.

### Interface Contracts

No external contracts — internal abstraction only.
