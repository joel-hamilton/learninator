# Implementation Plan: Collapse Generator Duplication

**Branch**: `023-collapse-generator-duplication` | **Date**: 2026-06-19 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `/specs/023-collapse-generator-duplication/spec.md`

## Summary

Collapse four structurally-identical lesson generation methods (`generateNext`, `generateSubLesson`, `generateRegenerate`, `generateBridging`) in `src/lessons/generator.ts` into a single `runGeneration(config: GenerationConfig)` template method. Each generation kind becomes a config object with `buildJobKey()`, `buildSystemPrompt()`, `buildUserMessage()`, and `findResult()` callbacks. The four public methods become one-line delegations to the template. No behavioral changes, no test modifications. Only `src/lessons/generator.ts` is modified.

## Phase 1: Design & Contracts

### Data Model

The key design artifact is the `GenerationConfig` interface:

```typescript
interface GenerationConfig {
  jobKeyType: "next" | "sub" | "regenerate" | "bridge";
  buildSystemPrompt: (missionId: number, mission: { title: string; status: string }, lesson: { number: number; subNumber: number | null; title: string }, direction?: "harder" | "easier") => string;
  buildUserMessage: (lesson: { number: number; subNumber: number | null; title: string }, opts?: { feedback?: string; notes?: string }, direction?: "harder" | "easier") => string;
  findResult: (missionId: number, lesson: { number: number; subNumber: number | null; title: string }) => Promise<{ lessonNumber: number; lessonSubNumber: number | null; lessonTitle: string } | null>;
}
```

Each of the four generation kinds maps to a concrete config object that implements these callbacks. The existing `buildJobKey` helper function at module level is reused by the config's `jobKeyType` field.

### Contracts

No external-facing contracts. The refactoring is purely internal to `LessonGenerator`. The public API surface (four public methods, `getJobStatus`, `JobStatus` type) is unchanged.
